# Tauri Billiards Wrapper — Specification

## Summary

Create a minimal Tauri v2 desktop application that wraps an existing hosted Three.js billiards game by loading its production URL in a desktop window. The project is a **new, standalone repo** — separate from the game source code. Phase 1 focuses on a working local build on Ubuntu. Phase 2 adds CI for Linux + Windows builds via GitHub Actions.

## Technical Decisions

### Tauri version
- **Tauri v2** (current stable major version)
- Installed via `cargo install tauri-cli --version "^2"`

### CLI management
- **Cargo only** — no Node.js, no npm, no `@tauri-apps/cli`
- Commands: `cargo tauri dev`, `cargo tauri build`

### Remote URL loading
- URL is set in **`tauri.conf.json` → `build.frontendDist`** as the remote URL string
- This is the single configurable location — change this one field to swap to a local `../dist` path later
- No environment variables, no Rust constants — keep it dead simple

### Rust code
- **Pure boilerplate only** — `tauri::Builder::default().run(tauri::generate_context!())`
- No custom commands, no plugins, no IPC, no menus
- No special error/loading handling — if the URL fails to load, the webview shows its default behavior (blank or browser error page). Keep it as simple as possible.

### Window configuration

| Property | Value |
|---|---|
| Title | `Billiards` |
| Width | `1024` |
| Height | `768` |
| Resizable | `true` |
| Fullscreen | `false` (start windowed; user can toggle via F11) |
| URL | `https://billiards.tailuge.workers.dev/lobby` (via `build.frontendDist`) |

### Security / CSP
- **CSP left unset** — use Tauri's default behavior
- No custom Content Security Policy configured
- Rationale: simplest approach, least risk of breaking the game's scripts, styles, or WebGL canvas. The game is served from the user's own Cloudflare Workers domain.

### Capabilities
- Minimal `capabilities/default.json` with `core:default` permissions only
- Must include a `remote` field listing `https://billiards.tailuge.workers.dev` so Tauri's IPC is available to the remote content (even though we use no custom IPC, Tauri v2 requires this for remote-loaded pages to function properly)

### Icons
- **User will provide their own** icon files in `src-tauri/icons/`
- For initial local builds, the tauri-cli may generate default placeholder icons during `cargo tauri init`, or we may need to create minimal placeholders so `cargo tauri build` doesn't fail
- Spec note: if `cargo tauri build` requires icons and none are provided, generate the Tauri default icon set via `cargo tauri icon` or use a simple placeholder

## Ubuntu Development Setup (from scratch)

### Step 1: Install system dependencies
```bash
sudo apt-get update
sudo apt-get install -y \
    libwebkit2gtk-4.1-dev \
    libgtk-3-dev \
    libayatana-appindicator3-dev \
    librsvg2-dev \
    patchelf \
    xdg-utils \
    build-essential \
    pkg-config \
    curl \
    file
```

**Note:** Tauri v2 requires **WebKitGTK 4.1** (not 4.0 used by Tauri v1). On Ubuntu 22.04+ this should be available. On older Ubuntu versions, may need a PPA or upgrade. We use `libayatana-appindicator3-dev` (the modern replacement for the legacy `libappindicator3-dev`).

### Step 2: Install Rust
```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source "$HOME/.cargo/env"
```

### Step 3: Install Tauri CLI
```bash
cargo install tauri-cli --version "^2"
```

### Step 4: Run the app
```bash
cargo tauri dev
```

### Step 5: Build locally (Linux)
```bash
cargo tauri build
```
Output binaries will be in `src-tauri/target/release/`.

## Limitations of Remote URL Approach

| Aspect | Remote URL (this approach) | Bundled Assets (future) |
|---|---|---|
| **Offline use** | ❌ Requires internet connection | ✅ Works offline |
| **Load speed** | Network-dependent | Instant (local files) |
| **Version pinning** | Game updates automatically (could introduce breaking changes) | Desktop app pinned to specific game version |
| **App store review** | May be rejected for loading external content | More likely to pass review |
| **CORS/security** | Depends on remote server headers | No CORS issues |
| **Initial startup** | Brief blank window until network loads | Instant content |
| **Error states** | No graceful offline handling in phase 1 | No network dependency |

## Phase 2: CI — Linux + Windows Builds — 📋 SPEC'D (not yet implemented)

### In scope
- GitHub Actions workflow that builds the Tauri app on **Linux, Windows, and macOS**
- Upload build artifacts (installers + binaries) as workflow artifacts — no GitHub Release created
- Manual dispatch (`workflow_dispatch`) and tag push (`v*`) triggers only — no branch push or PR checks
- Plain cargo commands (no `tauri-apps/tauri-action`, no Node.js dependency in CI)

### Out of scope
- GitHub Releases (deferred — artifacts only for now)
- Branch push / PR CI checks
- Auto updater
- Code signing (macOS builds will be unsigned — fine for personal use, not for distribution)

### CI Approach: Plain Cargo Commands

Since this is a **cargo-only project** (no `package.json`, no Node.js frontend), we use plain `cargo tauri build` commands instead of `tauri-apps/tauri-action`. This avoids needing Node.js, npm, or workarounds like `tauriScript`.

**Rationale:** `tauri-apps/tauri-action` is designed for projects with a Node frontend and expects npm/yarn/pnpm. For a cargo-only remote-URL wrapper, plain cargo commands are simpler and more reliable. Artifacts are uploaded via `actions/upload-artifact@v4`.

### Workflow Design

**File:** `.github/workflows/build.yml`

**Triggers:**
- `workflow_dispatch` (manual run from GitHub Actions tab)
- `push` on tags matching `v*` (future tagged releases)

**Strategy:** Matrix with `fail-fast: false`
- `ubuntu-22.04` (Linux)
- `windows-latest` (Windows)
- `macos-latest` (macOS)

**Steps per platform:**

1. **Checkout** — `actions/checkout@v4`
2. **Install Rust** — `dtolnay/rust-toolchain@stable`
3. **Rust cache** — `swatinem/rust-cache@v2` (keyed to `src-tauri`)
4. **Install Tauri CLI** — `cargo install tauri-cli --version "^2"`
5. **Install system deps (Linux only)** — `apt-get install` webkit2gtk-4.1, gtk-3, etc. (Windows and macOS have webview pre-installed)
6. **Build** — `cargo tauri build`
7. **Upload artifacts** — `actions/upload-artifact@v4` with platform-specific names

### Linux-specific details

Ubuntu runner needs these system packages:
```bash
sudo apt-get update
sudo apt-get install -y \
    libwebkit2gtk-4.1-dev \
    libgtk-3-dev \
    libayatana-appindicator3-dev \
    librsvg2-dev \
    patchelf \
    xdg-utils \
    build-essential \
    pkg-config
```

**Note:** Use `libayatana-appindicator3-dev` (not `libappindicator3-dev`) in CI — the legacy package may not be available on newer Ubuntu runner images.

### Windows-specific details

- **No system deps needed** — WebView2 runtime is pre-installed on `windows-latest` runners
- Build produces `.msi` installer and `.exe` in `src-tauri/target/release/bundle/`

### macOS-specific details

- **No system deps needed** — WKWebView is built into macOS
- Build produces `.dmg` and `.app` in `src-tauri/target/release/bundle/`
- **No code signing** — builds will be unsigned. Unsigned macOS apps can be run locally by right-clicking → Open, but can't be distributed via the Mac App Store or to other users without Apple Developer ID signing + notarization. This is fine for personal use and CI artifact testing.

### Artifact paths

| Platform | Bundle output path | Artifact name |
|---|---|---|
| Linux | `src-tauri/target/release/bundle/` (`.deb`, `.rpm`, `.AppImage`) | `billiards-linux` |
| Windows | `src-tauri/target/release/bundle/` (`.msi`, `.exe`) | `billiards-windows` |
| macOS | `src-tauri/target/release/bundle/` (`.dmg`, `.app`) | `billiards-macos` |

### Permissions

- `contents: read` — sufficient for artifact upload only
- If GitHub Releases are added later, change to `contents: write`

### Implementation

Phase 2 requires creating exactly **one file**: `.github/workflows/build.yml`

No changes to the Tauri app code, config, or Rust source are needed — the CI simply builds the existing Phase 1 code on GitHub-hosted runners.

**Workflow YAML:**

```yaml
name: Build

on:
  workflow_dispatch:
  push:
    tags:
      - 'v*'

jobs:
  build:
    permissions:
      contents: read
    strategy:
      fail-fast: false
      matrix:
        include:
          - platform: ubuntu-22.04
            artifact-name: billiards-linux
          - platform: windows-latest
            artifact-name: billiards-windows
          - platform: macos-latest
            artifact-name: billiards-macos
    runs-on: ${{ matrix.platform }}

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Install Rust stable
        uses: dtolnay/rust-toolchain@stable

      - name: Rust cache
        uses: swatinem/rust-cache@v2
        with:
          workspaces: 'src-tauri -> target'

      - name: Install Tauri CLI
        run: cargo install tauri-cli --version "^2"

      - name: Install system dependencies (Linux only)
        if: matrix.platform == 'ubuntu-22.04'
        run: |
          sudo apt-get update
          sudo apt-get install -y \
            libwebkit2gtk-4.1-dev \
            libgtk-3-dev \
            libayatana-appindicator3-dev \
            librsvg2-dev \
            patchelf \
            xdg-utils \
            build-essential \
            pkg-config

      - name: Build Tauri app
        run: cargo tauri build

      - name: Upload artifacts
        uses: actions/upload-artifact@v4
        with:
          name: ${{ matrix.artifact-name }}
          path: src-tauri/target/release/bundle/
```

### Testing plan (Phase 2)

1. Push the workflow file to the repo
2. Trigger via **Actions tab → Run workflow** (manual dispatch)
3. Verify both Linux and Windows jobs complete successfully
4. Download both artifacts from the workflow run
5. Verify Linux artifact contains `.deb` / `.AppImage`
6. Verify Windows artifact contains `.msi` / `.exe`
7. Verify macOS artifact contains `.dmg` / `.app`
